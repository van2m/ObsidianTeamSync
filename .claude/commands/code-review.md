# /review - 代码审查（Opus 子 Agent）

## 命令说明
将本次代码变更提交给 Opus 模型进行代码审查。审查通过后才能执行后续的提交流程。

## 工作流程

### 1. 收集变更信息
- 运行 `git diff` 查看当前所有未提交的变更
- 运行 `git diff --cached` 查看已暂存的变更
- 确认当前在 `dev` 分支

### 2. 启动 Opus 代码审查
使用 Agent 工具，指定 `model: opus`，将变更代码提交给 Opus 审查。

审查 Prompt 要求 Opus 从以下维度审查：
- **安全性**：是否存在注入、XSS、敏感数据泄露等风险
- **逻辑正确性**：业务逻辑是否正确，边界条件是否处理
- **代码质量**：命名规范、代码结构、是否有重复代码
- **性能**：是否有明显的性能问题（N+1 查询、不必要的循环等）
- **兼容性**：是否影响现有功能，是否与上游原版代码的合并兼容

### 3. 处理审查反馈
- 如果审查**通过**（无问题或仅有建议性意见）→ 输出 `✅ 代码审查通过`，流程结束
- 如果审查**有问题**：
  1. 展示 Opus 的反馈意见给用户
  2. 主 Agent 根据反馈修改代码
  3. 修改完成后重新提交审查（回到步骤 2）
  4. 循环直到审查通过

### 4. 审查通过后
输出审查摘要，提醒用户执行 `/post-change` 完成提交。

## 小修改跳过规则
以下情况可以跳过 Opus 审查，主 Agent 直接判断通过：
- 仅改注释、文档、格式化
- 单行修复（typo、变量名）
- 纯样式调整（CSS/WXSS）
- 用户明确说"跳过审查"

## 注意事项
- 每次审查都应该包含完整的 diff，不要遗漏文件
- 审查记录应简洁明了，突出关键问题
- 如果变更涉及数据库 schema，必须特别关注迁移兼容性
- 如果变更涉及 WebSocket 协议，必须确认插件端和服务端消息格式一致
